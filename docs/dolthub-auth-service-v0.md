# DoltHub Auth Service V0

Date: 2026-04-07

## Summary

Replace Nango with a separate `dolthub-auth` service that owns:

- one stable opaque `connection_id` per Wasteland subject per tenant and
  environment
- encrypted DoltHub API key custody
- Wasteland-specific connection metadata
- a DoltHub-only authenticated proxy that injects the DoltHub
  `Authorization` header server-side

Wasteland keeps browser sessions and app logic. The browser sends the DoltHub
API key directly to `dolthub-auth` with a one-time short-lived connect token.
Wasteland never receives the raw key. All authenticated DoltHub traffic from
Wasteland goes through the auth service proxy.

This is intentionally not a generic integration platform. It is a narrow
service for the exact hosted Wasteland + DoltHub use case.

## Phase 1 Constraints

- one Wasteland deployment talks to one auth-service deployment in production
- `tenant_id` and `environment` are first-class data and auth dimensions from
  day one
- phase 1 service auth uses deployment-scoped HMAC request signing with a
  shared secret that is dual-rotatable and not a global superuser key
- before one auth service instance is shared across multiple Wasteland
  deployments, service auth must move to signed service JWTs or mTLS
- phase 1 supports only authenticated DoltHub REST and GraphQL traffic
- phase 1 explicitly excludes `doltremoteapi.dolthub.com`

## What The Current Code Requires

The current hosted path uses Nango for four concrete jobs:

- browser-side credential submission through a short-lived connect flow in
  `web/src/components/ConnectPage.tsx`
- credential lookup in `internal/hosted/nango.go`
- metadata storage for `rig_handle` and `wastelands`
- an authenticated DoltHub proxy path in `internal/hosted/proxy.go`

The hosted backend currently consumes the raw DoltHub API key in multiple
places:

- connect validation in `internal/hosted/server.go`
- bootstrap warmup in `internal/hosted/server.go`
- workspace client construction in `internal/hosted/resolver.go`
- fork creation and rig registration in `internal/hosted/fork_register.go`

The code already supports client construction from a preconfigured HTTP client:

- `backend.NewRemoteDBWithClient(...)`
- `remote.NewDoltHubProviderWithClient(...)`

That makes an HTTP proxy boundary the right migration seam.

## Hosted Endpoint Inventory

As of 2026-04-07, the hosted path clearly relies on these authenticated
DoltHub surfaces:

- `https://www.dolthub.com/api/v1alpha1`
  - `internal/backend/remote.go`
  - `internal/hosted/server.go`
  - `internal/hosted/fork_register.go`
- `https://www.dolthub.com/graphql`

- instrument the proxy and Wasteland hosted path in staging
- confirm zero authenticated hosted requests target `doltremoteapi.dolthub.com`
- block production cutover if any hosted flow still depends on that host

## Decisions Locked In

These were explicitly resolved with the user:

- hard secret boundary: Wasteland must never receive the raw DoltHub API key
- all authenticated DoltHub traffic goes through the auth service
- browser sends the DoltHub API key directly to the auth service
- Wasteland keeps full ownership of browser sessions
- a single connection object holds both encrypted credential material and
  Wasteland metadata
- `connection_id` is stable, opaque, and generated by the auth service
- one active DoltHub connection per subject per tenant and environment
- reconnect updates the credential in place and preserves `connection_id`
- browser connect uses one-time short-lived connect tokens
- browser connect is direct to the auth service on its own origin
- auth service uses no browser cookies
- post-connect metadata changes are server-to-server from Wasteland to the auth
  service
- auth service is the source of truth for connection metadata
- Postgres is the primary store
- production encryption is backed by KMS, with a local master-key backend for
  local or non-production installs
- logs and telemetry are aggressively redacted
- lifecycle audit stays in Postgres
- logout destroys only the Wasteland browser session, not the stored
  connection
- phase 1 does not require an explicit end-user “forget credentials” feature
- environment-specific operator overlays such as real KMS resource names, IAM
  principals, Railway/Doppler wiring, and incident runbooks live in the private
  infra repo rather than this public repo

## Normative Identifiers

### `connection_id`

- type: ULID
- format: 26-character Crockford Base32 string
- generated only by the auth service
- stable across reconnects and metadata mutations

### `subject_id`

`subject_id` is security-critical and must obey all of these rules:

- opaque internal Wasteland principal identifier
- immutable for the lifetime of the principal
- never derived from `rig_handle`, email, or display name
- never recycled or reassigned to another user
- derived by Wasteland from its authenticated server-side session state, not
  from browser-supplied fields

### `tenant_id` and `environment`

- `tenant_id` identifies the Wasteland deployment namespace
- `environment` identifies the deployment environment, for example `prod`,
  `staging`, or `dev`
- `tenant_id` is opaque, immutable, and non-recycled
- auth-service authorization checks always include `tenant_id + environment`
- browser connect tokens and stored connections are scoped by both fields

## Service Boundary

### Wasteland owns

- browser sessions and cookies
- hosted auth UX
- validation of Wasteland metadata and business rules
- app endpoints like connect, join, leave, settings, bootstrap
- choosing whether a request is public or requires authenticated DoltHub access

### DoltHub auth service owns

- `connection_id` creation and lookup
- encrypted DoltHub API key custody
- DoltHub API key validation on create and reconnect
- Wasteland connection metadata persistence
- authenticated DoltHub proxying
- connection status and minimal audit trail

### Browser never gets

- decrypted DoltHub API key
- auth-service cookies
- Wasteland service credentials

## High-Level Architecture

### Components

- `wasteland`
  - browser-facing app
  - session owner
  - service client for `dolthub-auth`
- `dolthub-auth`
  - Postgres-backed auth service
  - KMS-backed encryption
  - browser-facing connect endpoints
  - Wasteland-facing metadata and proxy endpoints
- `postgres`
  - connections
  - connect tokens
  - audit log
- `kms`
  - master key for envelope encryption

### Runtime flow

1. Browser asks Wasteland to begin connect.
2. Wasteland validates proposed metadata.
3. Wasteland asks `dolthub-auth` for a one-time connect token scoped to:
   - `tenant_id`
   - `environment`
   - `subject_id`
   - exact approved initial metadata
4. Wasteland stores a hash of the one-time `redeem_secret` in the browser
   session and returns both the connect token and redeem secret to the browser.
5. Browser sends the DoltHub API key, the connect token, the redeem secret,
   and the approved metadata directly to
   `dolthub-auth`.
6. `dolthub-auth` atomically claims the token, validates both token proofs,
   validates the key with DoltHub,
   and creates or updates the stable connection.
7. `dolthub-auth` returns `connection_id`.
8. Browser calls Wasteland with `connection_id`.
9. Wasteland verifies all of the following before updating session state:
   - the same browser session still holds the matching redeem-secret hash
   - the returned `connection_id` resolves server-to-server to the same
     `subject_id`
   - the returned `connection_id` is the active connection for that subject
10. Wasteland then updates its browser session and proceeds normally.

After connect, all metadata updates are Wasteland server -> auth service.

All authenticated DoltHub HTTP goes Wasteland -> auth service -> DoltHub.

## Trust And Authentication Model

### Browser to auth service

- stateless bearer-token requests only
- no cookies
- CORS allowlisted per `tenant_id + environment`
- only connect-token redemption is browser-facing in phase 1

### Wasteland to auth service

Phase 1 uses deployment-scoped HMAC request signing, not a single global
bearer secret.

Each configured service credential has:

- `key_id`
- `tenant_id`
- `environment`
- hashed secret material
- `status`: `current` or `next`
- `not_before`
- `not_after`

Rules:

- a service secret is valid only for its configured `tenant_id + environment`
- the auth service derives tenant and environment scope from the secret itself
- caller-supplied tenant and environment values, if sent, must match the
  secret-bound scope or the request is rejected
- requests must still include `subject_id` and `connection_id` where
  applicable
- the auth service verifies that the addressed connection belongs to the same
  `tenant_id + environment + subject_id`
- TLS is mandatory end-to-end
- phase 1 should also enforce a network allowlist between Wasteland and the
  auth service
- every request must include:
  - `X-Service-Key-Id`
  - `X-Service-Timestamp`
  - `X-Service-Nonce`
  - `X-Service-Body-SHA256`
- the auth service verifies an HMAC-SHA-256 signature over:
  - `X-Service-Key-Id`
  - HTTP method
  - normalized path
  - canonical query string
  - body SHA-256
  - timestamp
  - nonce
  - `X-Auth-Subject-Id`
  - `X-Auth-Connection-Id` when present
- canonical request rules are normative:
  - path normalization matches the proxy/path sanitizer rules
  - query parameters are percent-encoded, sorted by key then value
  - body hash is lowercase hex SHA-256 over the exact request bytes
  - timestamp is UTC Unix seconds
  - nonce must contain at least 128 bits of CSPRNG entropy
- client and server must share test vectors for canonicalization and signature
  verification before implementation is accepted
- unsigned identity headers must never be used for authorization decisions
- nonce reuse is rejected across all auth-service instances during the
  acceptance window
- requests are rejected if clock skew exceeds 15 seconds
- request-signing state must be stored in a shared nonce store, with Postgres
  acceptable in phase 1

### Dual-secret rotation

Phase 1 must support exactly two active secrets per deployment scope:

- `current`
- `next`

Rotation flow:

1. provision `next`
2. roll Wasteland to send `next`
3. keep `current` accepted during the grace window
4. retire `current`

### Hardening path

Before one auth-service deployment serves multiple Wasteland deployments or
untrusted cross-network callers, phase 1 HMAC request signing must be replaced
by
one of:

- signed service JWTs with cryptographically bound `tenant_id + environment`
  claims
- mTLS with deployment-scoped client certs

This is a release gate for shared multi-deployment use.

## Connection Model

The primary resource is a single connection object.

### Connection identity

- `connection_id`
- `tenant_id`
- `environment`
- `subject_id`, which must be a UUIDv4, ULID, or another opaque identifier
  with at least 128 bits of entropy

Uniqueness:

- one active connection per `tenant_id + environment + subject_id`

Reconnect behavior:

- preserve `connection_id`
- replace encrypted credential in place
- increment `credential_version`
- preserve metadata

### Metadata on the connection

- `rig_handle`
- `wastelands`
- connection health fields
- timestamps

`wastelands` remains an array in phase 1 because it matches current hosted
behavior: one credential can back multiple joined wastelands.

## Persistence Model

Use Postgres from day one.

### DDL

```sql
CREATE TABLE connections (
    connection_id CHAR(26) PRIMARY KEY,
    tenant_id TEXT NOT NULL,
    environment TEXT NOT NULL,
    subject_id TEXT NOT NULL,
    rig_handle TEXT NOT NULL,
    wastelands JSONB NOT NULL DEFAULT '[]'::jsonb,
    status TEXT NOT NULL CHECK (status IN ('active', 'invalid', 'degraded')),
    credential_ciphertext BYTEA NOT NULL,
    credential_key_version TEXT NOT NULL,
    credential_encryption_backend TEXT NOT NULL,
    credential_version INTEGER NOT NULL DEFAULT 1,
    record_version INTEGER NOT NULL DEFAULT 0,
    last_validated_at TIMESTAMPTZ,
    last_validation_error_code TEXT CHECK (
        last_validation_error_code IN (
            'invalid_key',
            'expired_key',
            'revoked_key',
            'upstream_unreachable',
            'rate_limited',
            'kms_unavailable',
            'proxy_unauthorized'
        )
    ),
    last_proxy_error_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (tenant_id, environment, subject_id),
    CHECK (jsonb_typeof(wastelands) = 'array')
);

CREATE INDEX connections_subject_lookup_idx
    ON connections (tenant_id, environment, subject_id);

CREATE TABLE connect_tokens (
    connect_token_mac BYTEA PRIMARY KEY,
    redeem_secret_mac BYTEA NOT NULL,
    tenant_id TEXT NOT NULL,
    environment TEXT NOT NULL,
    subject_id TEXT NOT NULL,
    approved_metadata JSONB NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    used_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX connect_tokens_reap_idx
    ON connect_tokens (expires_at)
    WHERE used_at IS NULL;

CREATE TABLE service_request_nonces (
    key_id TEXT NOT NULL,
    nonce TEXT NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (key_id, nonce)
);

CREATE INDEX service_request_nonces_reap_idx
    ON service_request_nonces (expires_at);

CREATE TABLE audit_log (
    audit_id BIGSERIAL PRIMARY KEY,
    occurred_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    tenant_id TEXT NOT NULL,
    environment TEXT NOT NULL,
    subject_id TEXT NOT NULL,
    connection_id CHAR(26),
    actor_type TEXT NOT NULL,
    action TEXT NOT NULL,
    outcome TEXT NOT NULL,
    error_code TEXT,
    metadata JSONB NOT NULL DEFAULT '{}'::jsonb
);

CREATE INDEX audit_log_scope_idx
    ON audit_log (tenant_id, environment, occurred_at DESC);
```

### Transaction and concurrency rules

- `READ COMMITTED` is sufficient in phase 1 because correctness relies on
  unique constraints, row-level updates, and compare-and-swap semantics
- no database transaction is held open across DoltHub or KMS network calls
- all metadata mutations must update `record_version = record_version + 1`
  under `WHERE record_version = $expected`
- reconnect must update `credential_version = credential_version + 1`
- token redemption must be atomic and verify both browser proofs:

```sql
UPDATE connect_tokens
SET used_at = now()
WHERE connect_token_mac = $1
  AND redeem_secret_mac = $2
  AND used_at IS NULL
  AND expires_at > now()
RETURNING tenant_id, environment, subject_id, approved_metadata;
```

If the update returns zero rows, the token is invalid, expired, or already
used.

### Connect-token lifecycle

- tokens are one-time use only
- default TTL: 5 minutes
- cap outstanding unredeemed tokens at 3 per
  `tenant_id + environment + subject_id`
- expired or used tokens are deleted by a reap job every 15 minutes
- tokens older than 24 hours are purged unconditionally

### Audit retention

- audit rows are append-only
- normal successful proxy traffic is not stored in `audit_log`
- retain audit rows for 180 days
- phase 1 may use a single table; monthly partitioning is the first expansion
  step if volume grows
- `audit_log.metadata` must use a fixed schema with size caps; arbitrary
  request-derived blobs or exception objects are forbidden

## Encryption Model

### Production

- envelope encryption
- master key in cloud KMS
- ciphertext and encryption metadata in Postgres
- plaintext exists only in process memory during validation or proxy use
- KMS decrypt permission must be granted only to the auth-service runtime role
- Postgres backup or read-only operators must not also hold KMS decrypt rights
- KMS key rotation uses an explicit background re-encryption job; decrypt paths
  do not silently rewrite stored rows

### Local or non-production portability

- pluggable master-key backend
- local master key from environment variable or secret file
- explicitly non-production by default
- startup must fail closed if `environment = prod` and the configured
  encryption backend is not KMS-backed

### Caching

Phase 1 removes decrypted-credential caching entirely.

Rationale:

- reconnect must stop using the old credential immediately
- cross-instance cache invalidation is not worth the risk in phase 1
- KMS decrypt on each authenticated request is acceptable at initial scale

If future load requires caching, the first allowed design is a version-keyed
cache on `(connection_id, credential_version)` with immediate invalidation on
credential update.

## Browser-Facing API

These endpoints live on the auth-service origin and are CORS-allowlisted to
approved Wasteland browser origins for the matching tenant and environment.

No cookies are used.

### `POST /v1/connect-tokens/redeem`

Purpose:

- browser submits the DoltHub API key directly to the auth service

Auth:

- one-time `connect_token`
- one-time `redeem_secret`

Request body:

```json
{
  "connect_token": "base64url-token",
  "redeem_secret": "base64url-secret",
  "api_key": "dolt-api-key",
  "metadata": {
    "rig_handle": "alice",
    "wastelands": [
      {
        "upstream": "hop/wl-commons",
        "fork_org": "alice-org",
        "fork_db": "wl-commons",
        "mode": "pr",
        "signing": true
      }
    ]
  }
}
```

Behavior:

- verify HMAC-SHA-256 of the connect token using a server-side token pepper
- verify HMAC-SHA-256 of the redeem secret using a separate server-side pepper
- atomically claim the token using both proofs
- derive `tenant_id + environment` from the redeemed token record, not from
  caller input
- require the browser `Origin` header to match the exact allowlist configured
  for that token-scoped `tenant_id + environment`
- reject redemption if `Origin` is absent or outside that exact scope
- canonicalize request metadata and compare it byte-for-byte with the approved
  metadata bound to the token
- validate the DoltHub API key with the same lightweight authenticated query
  shape Wasteland uses today, equivalent to `SELECT 1` on the REST SQL API
- treat only explicit DoltHub credential-rejection responses as invalid-key
  outcomes; temporary upstream failures remain retryable service errors
- create or update the stable connection

Success response:

```json
{
  "connection_id": "01HS8Z6Q6F8M0A8Z3S2R4D5T6U",
  "status": "active",
  "last_validated_at": "2026-04-07T19:00:00Z"
}
```

Error response:

```json
{
  "error_code": "invalid_api_key",
  "user_message": "That DoltHub API key was rejected. Check the key and try again.",
  "retryable": false,
  "request_id": "req_123"
}
```

Allowed `error_code` values:

- `invalid_connect_token`
- `expired_connect_token`
- `used_connect_token`
- `metadata_mismatch`
- `invalid_api_key`
- `expired_api_key`
- `revoked_api_key`
- `upstream_unreachable`
- `rate_limited`
- `service_unavailable`

### Connect-token cryptography

- token entropy: at least 256 bits from a CSPRNG
- token format: opaque base64url string
- redeem-secret entropy: at least 256 bits from a CSPRNG
- storage: HMAC-SHA-256 with server-side peppers and explicit domain
  separation labels
- comparison: constant-time comparison on the MAC result
- metadata binding: RFC 8785 JSON Canonicalization Scheme
- `connect_token` is delivered only in JSON request bodies
- `redeem_secret` is delivered only in JSON request bodies
- `connect_token` must never appear in URLs, browser storage, referrers, or
  frontend telemetry
- `redeem_secret` must never appear in URLs, browser storage, referrers, or
  frontend telemetry

## Wasteland-Facing API

These endpoints are service-to-service only.

Auth:

- `X-Service-Key-Id: <key-id>`
- `X-Service-Timestamp: <unix-seconds>`
- `X-Service-Nonce: <opaque-random-nonce>`
- `X-Service-Body-SHA256: <hex>`
- `Authorization: HMAC-SHA256 <signature>`
- `X-Auth-Subject-Id: <subject-id>`
- `X-Auth-Connection-Id: <connection-id>` where applicable

The auth service derives `tenant_id + environment` from the service credential
scope. If the caller also sends them, they are checked for equality and ignored
as a source of truth.

### `POST /v1/connect-tokens`

Purpose:

- mint one-time short-lived connect token for a browser connect flow

Request:

- `subject_id`
- exact approved initial metadata payload
- optional TTL override within the hard maximum

Response:

- `connect_token`
- `redeem_secret`
- `expires_at`

### `GET /v1/connections/{connection_id}`

Purpose:

- retrieve non-secret connection metadata and status

Authorization:

- allowed only if the resolved row matches the secret-bound
  `tenant_id + environment`, the signed `X-Auth-Subject-Id`, and the path
  `connection_id`
- if `X-Auth-Connection-Id` is present, it must exactly equal the path
  `connection_id`

Response:

- `connection_id`
- `subject_id`
- `rig_handle`
- `wastelands`
- `status`
- `last_validated_at`
- `last_validation_error_code`
- `created_at`
- `updated_at`
- `record_version`

### `GET /v1/subjects/{subject_id}/connection`

Purpose:

- retrieve the current active connection for a subject
- supports session/bootstrap repair if Wasteland needs to recover
  `connection_id` server-side

Authorization:

- path `subject_id` must exactly equal `X-Auth-Subject-Id`
- the returned row must also match the secret-bound `tenant_id + environment`

Response:

- `connection_id`
- the same non-secret summary returned by
  `GET /v1/connections/{connection_id}`

### `PATCH /v1/connections/{connection_id}/rig-handle`

Purpose:

- update `rig_handle`

### `PUT /v1/connections/{connection_id}/wastelands/{upstream}`

Purpose:

- upsert one wasteland entry

### `DELETE /v1/connections/{connection_id}/wastelands/{upstream}`

Purpose:

- remove one wasteland entry

### `PATCH /v1/connections/{connection_id}/wastelands/{upstream}/settings`

Purpose:

- update per-wasteland settings like `mode` and `signing`

### Metadata mutation semantics

- targeted patch-style operations, not blind full-object replacement
- every write requires `record_version`
- conflicting writes return `409 conflict`
- browser never writes metadata directly after initial connect
- any identity-bearing path parameter must exactly match the corresponding
  signed identity header or the request is rejected before lookup or mutation

## Proxy API

The proxy is DoltHub-only and handles authenticated traffic only.

### Allowed upstreams

Phase 1 allowlist:

- `https://www.dolthub.com/api/v1alpha1`
- `https://www.dolthub.com/graphql`

Not included in phase 1:

- `https://doltremoteapi.dolthub.com`

### Allowed methods

Phase 1:

- `GET`
- `POST`
- `PATCH`
- `DELETE` is not allowed in phase 1

Always reject:

- arbitrary hosts
- `CONNECT`
- `TRACE`
- hop-by-hop header forwarding

### Route shape

Use fixed DoltHub surfaces rather than arbitrary destination URLs:

- `/v1/proxy/api/{path...}` ->
  `https://www.dolthub.com/api/v1alpha1/{path...}`
- `/v1/proxy/graphql` -> `https://www.dolthub.com/graphql`

Redirect policy:

- the proxy must not follow upstream redirects

### Path sanitization

For `/v1/proxy/api/{path...}`:

- reject `..`
- reject backslashes
- reject encoded slashes that would alter path segment boundaries
- normalize before constructing the upstream URL
- if normalization changes the semantic path, reject the request

### Proxy behavior

- verify service auth
- verify `tenant_id + environment + subject_id + connection_id`
- look up and decrypt the connection credential
- after decrypt, re-check `credential_version` once before proxying; if it
  changed, discard the plaintext and retry the lookup one time
- inject `Authorization: token <apiKey>`
- strip any caller-supplied `Authorization`
- strip any caller-supplied `Proxy-Authorization`, `Cookie`, `Set-Cookie`,
  `X-Api-Key`, `Forwarded`, `Via`, and `X-Forwarded-*`
- forward only this request-header allowlist:
  - `Accept`
  - `Content-Type`
  - `User-Agent`
  - `If-Match`
  - `If-None-Match`
  - `X-Request-Id`
- framing headers like `Content-Length` and transfer encoding are regenerated
  by the proxy client stack and are never forwarded from the caller
- stream request and response bodies by default
- do not buffer full bodies unless needed
- do not automatically retry write methods
- preserve upstream response status and body when safe
- add `X-Wasteland-Auth-Error-Code` on auth-service-generated failures and on
  normalized upstream credential failures
- strip hop-by-hop response headers and `Set-Cookie`
- auth-service-generated error responses must never echo upstream request
  headers, request bodies, or decrypted connection details
- raw upstream response headers and bodies must never be attached to generic
  logs, traces, exception reports, or audit metadata before normalization
- passthrough bodies are allowed only for vetted DoltHub content types and
  bounded sizes; otherwise the proxy returns a normalized error envelope
- outbound transport policy is normative:
  - ignore `HTTP_PROXY`, `HTTPS_PROXY`, and similar env-configured forward
    proxies
  - use standard TLS hostname validation and SNI for the two allowed DoltHub
    hosts only
  - require network egress allowlisting to those upstream hosts

### Proxy status mapping

- upstream response that explicitly signals credential rejection:
  - return upstream status
  - set `X-Wasteland-Auth-Error-Code: invalid_key`
  - re-read `credential_version`; only if it still matches the version used
    for the request may the service set `status = 'invalid'`
  - only then set `last_validation_error_code = 'invalid_key'`
- plain upstream `403` without explicit invalid-credential evidence:
  - return upstream status
  - do not mark the credential invalid
- auth-service policy failure:
  - return JSON error envelope
  - use `403` for ownership mismatch
  - use `429` for local rate limits
  - use `503` or `504` for dependency failures

## Connection Health Model

Expose non-secret status on each connection:

- `status`
  - `active`
  - `invalid`
  - `degraded`
- `last_validated_at`
- `last_validation_error_code`
- `last_proxy_error_at`

Allowed `last_validation_error_code` values:

- `invalid_key`
- `expired_key`
- `revoked_key`
- `upstream_unreachable`
- `rate_limited`
- `kms_unavailable`
- `proxy_unauthorized`

Behavior:

- create and reconnect validate the key before accepting it
- invalid reconnect leaves the existing active credential untouched
- temporary upstream failures set `status = 'degraded'`, not `invalid`
- Wasteland uses this status surface for bootstrap and reconnect UX

## Timeouts, Backpressure, And Degradation

### Timeout budget table

| Operation | Timeout |
| --- | --- |
| Postgres query timeout | 250 ms |
| Postgres transaction timeout | 1 s |
| KMS encrypt or decrypt | 750 ms |
| DoltHub validation during connect/reconnect | 5 s total |
| Proxy upstream connect timeout | 2 s |
| Proxy upstream response-header timeout | 10 s |
| Proxy total request timeout | 20 s for `GET`, 30 s for `POST`/`PATCH`/`DELETE` |
| Wasteland call to auth-service metadata APIs | 3 s |
| Wasteland call to auth-service proxy APIs | slightly above the proxy total timeout |

### Concurrency and pool limits

- per-instance upstream connection pool cap per DoltHub host
- per-`connection_id` concurrency cap to avoid one user saturating the proxy
- global in-flight proxy request cap per auth-service instance
- above-limit requests fail fast with `429` rather than queueing indefinitely

Phase 1 default sizing:

- target steady state: 20 authenticated proxy RPS per instance
- target burst: 100 authenticated proxy requests in flight per instance
- if observed production load exceeds this materially, add version-keyed
  decrypted-credential caching before raising the cap

### Failure-mode matrix

| Failure | Auth-service behavior | Wasteland behavior |
| --- | --- | --- |
| Postgres unavailable | readiness false, return `503 service_unavailable` | keep browser session, fail authenticated ops closed |
| KMS unavailable | readiness false, return `503 kms_unavailable` | keep browser session, show temporary-service error |
| DoltHub slow or unreachable | timeout or `502/504`, set connection `degraded` | keep browser session, show temporary-service error |
| Auth service overload | `429` or `503` | keep browser session, fail authenticated ops closed |
| Invalid DoltHub credential | `401/403` with `X-Wasteland-Auth-Error-Code: invalid_key` | prompt reconnect, do not log user out |

## Session And Lifecycle Semantics

### Logout

- destroys only the Wasteland browser session
- does not delete the connection
- does not touch stored DoltHub credentials

### Leave wasteland

- metadata mutation only
- does not delete the connection

### Explicit credential deletion

Phase 1 does not need a user-facing delete-credential flow.

If added later, it must be an explicit action like:

- “Forget DoltHub credentials”
- “Remove stored connection”

and never a side effect of logout.

## Security And Policy Defaults

### CORS

- explicit allowlist per tenant and environment
- no wildcard origins
- no credentials mode
- no auth-service cookies

### Logging

Never log:

- API key payloads
- request bodies on connect or proxy routes
- upstream `Authorization` headers
- raw rewritten header sets
- raw query strings by default
- connect-token or redeem-secret values

Ingress and middleware policy:

- the same secret-safe logging policy applies to the Wasteland endpoint that
  returns `connect_token` and `redeem_secret`
- no request mirroring on connect or proxy routes
- no access-log body capture on connect or proxy routes
- no APM header or body sampling on connect or proxy routes
- no crash reporting that serializes request bodies or auth headers on those
  routes
- these are deployment requirements, not conventions, and must be validated in
  staging before production cutover
- production cutover also requires a secret-safe observability conformance
  suite covering ingress logs, request mirroring, APM, tracing, and crash
  reporting on connect and proxy routes
- if the required safe-observability controls are not enabled, production
  startup must fail closed

Log only:

- route
- method
- status
- latency
- tenant
- environment
- error code
- `connection_id` in logs and traces only when needed

### Telemetry

- metrics:
  - request count
  - latency
  - error count
- labels:
  - route
  - method
  - status class
  - upstream host class
  - environment
- exclude `connection_id` from metrics labels
- treat `subject_id` and `connection_id` as sensitive operational identifiers
- allow them in logs and traces only when needed for debugging
- forbid them from arbitrary audit metadata payloads

Trace and middleware enforcement:

- connect and proxy routes default-deny tracing attributes for:
  - request bodies
  - request headers
  - query strings
  - `subject_id`
  - `connection_id`
- any break-glass debug mode must be explicit, time-bounded, and unavailable by
  default in production
- browser redemption must pass an edge conformance test suite that covers exact
  `Origin` handling, absent `Origin`, malformed `Origin`, and real preflight
  behavior through the production-like CDN/LB/proxy path

### Rate limits

Separate budgets for:

- browser connect-token redemption and credential submission
- Wasteland metadata patch calls
- Wasteland proxy calls

### Health endpoints

- liveness: process only
- readiness: require Postgres and KMS
- readiness does not require DoltHub to be reachable

## Backup And Disaster Recovery

- Postgres backups must remain encrypted at rest
- the auth-service database must never be restored into a lower environment
  with production KMS access
- disaster recovery requires both:
  - the Postgres backup
  - the relevant KMS key versions
- restore drills should validate that encrypted credentials remain decryptable
  only in the intended environment
- restore drills should also validate that connect-token peppers and HMAC
  signing secrets are not present in lower environments

## UX State Model

- first connect:
  - browser receives connect token from Wasteland
  - browser submits API key directly to auth service
  - browser sends returned `connection_id` back to Wasteland
- returning authenticated session:
  - Wasteland uses the stored `connection_id`
  - if needed, Wasteland can recover the active connection via
    `GET /v1/subjects/{subject_id}/connection`
- invalid credential:
  - Wasteland keeps browser session
  - user sees reconnect-required UX
- temporary auth-service or DoltHub outage:
  - Wasteland keeps browser session
  - user sees a temporary-service error, not a reconnect prompt

## Migration Shape

### 1. Introduce auth-service client in Wasteland

- add a client abstraction replacing `NangoClient`
- add service config:
  - base URL
  - `key_id`
  - bearer secret
  - tenant
  - environment

### 2. Replace browser connect flow

- replace Nango frontend SDK calls in `web/src/api/nango.ts`
- browser receives connect token from Wasteland
- browser posts directly to auth service
- browser sends resulting `connection_id` back to Wasteland

### 3. Replace hosted metadata reads and writes

- connect
- auth status
- bootstrap
- join
- leave
- settings save

### 4. Replace authenticated DoltHub clients

- stop constructing token-based clients in hosted mode
- build hosted `RemoteDB` and `DoltHubProvider` with HTTP clients whose
  transport targets the new auth proxy

### 5. Migrate fork registration explicitly

`internal/hosted/fork_register.go` currently takes a raw `apiKey`. That must be
rewritten before cutover.

Required design:

- change fork registration to use proxy-backed clients
- construct `remote.NewDoltHubProviderWithClient(...)` and
  `backend.NewRemoteDBWithClient(...)`
- remove raw-key parameters from the hosted fork registration interface

### 6. Fix hosted caches and environment assumptions

- `pendingUpstreamCache` must not be shared only by upstream in hosted mode
- key it by `connection_id + upstream`, or disable shared cross-user caching in
  phase 1
- hosted mode must not rely on `DOLTHUB_SESSION_TOKEN`
- both Wasteland and the auth service must run with `DOLTHUB_SESSION_TOKEN`
  unset in hosted deployments

### 7. Remove Nango integration

- delete Nango env vars and wiring
- delete Nango frontend dependency
- delete Nango metadata client code and tests

## Main Risks

- the proxy becomes a new critical-path dependency for all authenticated hosted
  DoltHub traffic
- fork registration and workspace resolution both require careful migration from
  raw-key clients to proxy-backed clients
- GraphQL and REST host pinning is simple, but production cutover must prove no
  hosted dependency still needs `doltremoteapi.dolthub.com`
- JSONB `wastelands` metadata requires strict CAS semantics to avoid lost
  updates

## Recommended Phase 1 Scope

Implement only:

- connect token minting
- direct browser credential submission
- connection create or reconnect with validation
- connection metadata read and patch operations
- DoltHub REST and GraphQL authenticated proxy
- Postgres store
- KMS-backed envelope encryption
- minimal audit and telemetry

Do not implement in phase 1:

- generic provider support
- user-facing credential deletion UX
- admin UI
- `doltremoteapi.dolthub.com` support
- decrypted-credential cache
- distributed cache invalidation
- event-pipeline audit fanout

## Bottom Line

The smallest clean replacement for Nango here is not another integration
vendor. It is a dedicated `dolthub-auth` service that:

- stores one encrypted DoltHub credential per Wasteland subject
- stores Wasteland connection metadata on the same object
- accepts browser-side credential submission directly
- exposes a DoltHub-only authenticated proxy to Wasteland

This version closes the major design gaps from the first review pass by making
the invariants, migration edges, and failure contracts explicit enough to
implement against.
